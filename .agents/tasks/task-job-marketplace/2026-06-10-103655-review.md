# Full-stack job marketplace with deterministic matching engine

A Next.js 16 application implementing a two-sided job marketplace (BDJobs AI) with Supabase as the backend. The implementation covers auth, role-based dashboards for job seekers and employers, a deterministic scoring engine that rates candidate-job fit across five weighted dimensions, an application workflow with state-machine transitions, and real-time messaging gated behind application acceptance. The architecture is server-action-heavy with RLS policies as a second enforcement layer at the database.

**Watch for:** Sign-out is broken — the navbar posts to a non-existent `/api/auth/logout` route (confirmed). Message pagination has an off-by-one that leaks an extra message per page (confirmed). The conversation list fires 2N+1 queries for N conversations (confirmed). Role enforcement in server actions relies solely on profile lookups, not on a session claim (likely).

## High-level view

Authentication uses Supabase email/password with a `proxy.ts` (Next.js 16's replacement for middleware) that redirects unauthenticated users away from `/dashboard`. The logout mechanism is wired incorrectly: the navbar submits a form to `/api/auth/logout` which does not exist, while the actual `logout()` server action in `auth.ts` is never referenced from the UI.

The matching engine is a deterministic weighted scorer with five dimensions (30% degree, 30% skills, 20% category, 15% experience, 5% language) and a hard cap at 20% for fresh graduates applying to senior roles.

Authorization flows through server actions that verify ownership at every step (user owns company, company owns job, job owns application). However, the RLS policy on the `users` table allows any authenticated user to update any column on their own row — including `role` — creating a privilege escalation path.

Real-time messaging subscribes to Postgres changes via Supabase Realtime channels. The conversation list page degrades linearly with conversation count due to per-conversation queries for last message and unread count (2N+1 queries total).

1. **Sign-out form targets missing route** — The navbar posts to `/api/auth/logout` which doesn't exist. Wire the form to invoke the `logout()` server action instead.
2. **Message pagination off-by-one** — `getMessages` uses `.range(offset, offset + pageSize)` (inclusive both ends = pageSize+1 items) but returns all fetched items. Either use `.range(offset, offset + pageSize - 1)` or slice the result array before returning.
3. **Conversation list N+1 queries** — `getConversations` executes 2 additional queries per conversation (last message + unread count). Batch these into aggregate queries or use a database view/function.
4. **Role not enforced via session claims** — Server actions determine role by querying the `users` table. A user who modifies their `role` column (if RLS allows `UPDATE` on that field) could escalate privileges. The RLS `UPDATE` policy on `users` allows updating any field on one's own row.
5. **Seed migration runs in production path** — `00004_seed_data.sql` lives alongside schema migrations with no environment guard. It will execute on any Supabase instance that runs migrations.
6. **No input sanitization on ilike filter** — `getJobs` interpolates `filters.search` directly into `.or()` and `filters.location` into `.ilike()`. Supabase parameterizes these, but special characters (`%`, `_`) in user input will behave as wildcards.
7. **Realtime subscription not scoped to accepted status** — The `subscribeToMessages` function subscribes to any INSERT on the messages table filtered only by `conversation_id`. If RLS is misconfigured or bypassed, messages could leak.

## Broken sign-out flow

The navbar component renders a `<form action="/api/auth/logout" method="POST">` in both desktop and mobile menus. No route handler exists at that path — there's no `src/app/api/` directory at all. The `logout()` server action in `src/lib/actions/auth.ts` correctly calls `supabase.auth.signOut()` and redirects to `/login`, but nothing in the UI invokes it. Clicking "Sign Out" will produce a 404.

## Message pagination off-by-one

In `src/lib/actions/messages.ts`, `getMessages` does:

```typescript
const pageSize = 20
const offset = page * pageSize
// ...
.range(offset, offset + pageSize)
```

Supabase's `.range()` is inclusive on both bounds, so this returns up to 21 rows. The `hasMore` calculation (`messages.length > pageSize`) correctly detects when a 21st item exists, but all 21 items are returned to the client as actual messages rather than slicing off the sentinel row. Subsequent pages from the chat window start at `page = 1`, fetching offset 20..40 (items 20-40), which overlaps with the 21st item from the first call. The combination produces duplicate messages at pagination boundaries.

## Conversation list performance

`getConversations` fetches all conversations for the current user, then iterates and fires:
- One query for the last message (`messages` table, ordered, limit 1)
- One count query for unread messages

For a user with 50 conversations, that's 101 queries executed sequentially in a single server action call. This should be rewritten as a single query using Postgres subqueries or a database function that returns conversations with aggregated last-message and unread-count fields.

## Role escalation surface

The `users` table RLS policy allows authenticated users to update their own row:

```sql
CREATE POLICY "Users can update own profile"
  ON users FOR UPDATE
  TO authenticated
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);
```

No column restriction exists. A user could update their own `role` field from `'jobseeker'` to `'employer'` via a direct Supabase client call (the anon key is exposed client-side via `NEXT_PUBLIC_SUPABASE_ANON_KEY`). The server actions trust `profile.role` to determine access — employer actions check company ownership which requires being an employer. Adding a check constraint (`ALTER TABLE users ADD CONSTRAINT role_immutable CHECK (role = role)` won't work — use a trigger or column-level RLS via a restricted update policy) would close this.

## Seed data in migration path

`supabase/migrations/00004_seed_data.sql` inserts development users with hardcoded UUIDs alongside schema migrations with no environment guard. It will execute on any Supabase instance that runs migrations. The file inserts into `users` without corresponding `auth.users` entries (noted in a comment), meaning the foreign key `REFERENCES auth.users(id)` will fail unless those auth entries already exist — making this migration non-portable to any fresh instance.

## Search filter wildcard injection

In `src/lib/actions/jobs.ts`:

```typescript
if (filters.search) {
  query = query.or(`title.ilike.%${filters.search}%,description.ilike.%${filters.search}%`)
}
```

The `%` and `_` characters in user input are interpreted as LIKE wildcards. A search for "100%" would match any title containing "100" followed by any character. Minor UX issue rather than a security vulnerability, but worth escaping if search accuracy matters.

## Missing test coverage

There are no test files in the codebase. The matching engine's pure functions (`scoreDegreeRelevance`, `scoreSkillOverlap`, etc.) are the most testable part of the system and the most critical to get right — particularly the fresh-grad cap logic and the degree-field-to-category mapping. The RLS policies are also untested and contain the role escalation gap described above.
