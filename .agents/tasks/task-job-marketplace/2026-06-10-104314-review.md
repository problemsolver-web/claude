# Full-stack job marketplace with deterministic matching engine (v2)

The five critical issues from the first review have been addressed. Logout now invokes the `logout()` server action directly from form elements. Message pagination correctly uses a sentinel pattern (fetch pageSize+1, slice, detect hasMore). The role escalation surface is closed by both an application-level strip in `profile.ts` and a database trigger (`prevent_role_change`) that raises an exception on any role mutation. The conversation list replaced its per-conversation queries with two batched `.in()` calls. The seed migration gained a prominent development-only header with documentation about auth.users prerequisites.

**Watch for:** The batched "last message" query fetches every message row for all conversations without a limit, which degrades as message volume grows (confirmed). The `register` server action accepts any string for `role` from the client form without validating it against the enum (confirmed). The `createCompany` action has no role check, allowing a job seeker to create a company if they call the action directly (likely).

**Verdict**: APPROVED

## High-level view

The conversation list N+1 fix trades per-conversation queries for two bulk queries, which is a large improvement for typical conversation counts. However, the "recent messages" query has no row limit and pulls all messages across all conversations into application memory to deduplicate in JavaScript. For a user with 50 conversations averaging 200 messages each, that's 10,000 rows transferred and iterated.

Registration accepts the `role` field directly from the client `FormData` and passes it to the Supabase insert. The enum type at the database level will reject unexpected values, but there is no application-level validation that the value is one of the two allowed roles. More importantly, there is no server-side enforcement that prevents a job seeker from calling `createCompany` directly as a server action.

## Unbounded message fetch in conversation list

The N+1 fix replaces per-conversation queries with a single batched fetch:

```typescript
const { data: recentMessages } = await supabase
  .from('messages')
  .select('conversation_id, content, created_at')
  .in('conversation_id', conversationIds)
  .order('created_at', { ascending: false })
```

This query has no `.limit()` clause. Supabase's default limit is 1000 rows, but that's still far more than needed when only the latest message per conversation matters. For conversations with heavy message activity, this transfers and iterates through substantial data just to extract one row per conversation via the `lastMessageMap` deduplication loop.

A more efficient approach would be a Postgres function using `DISTINCT ON (conversation_id)` or adding a `.limit(conversationIds.length)` if the ordering guarantees are strong enough (they aren't here because multiple messages per conversation exist). The pragmatic fix is an RPC that returns one latest message per conversation in a single round-trip.

The unread-count query has the same shape but is less concerning because unread messages are typically a small subset.

## Role validation gap at registration

In `auth.ts`:

```typescript
const role = formData.get('role') as 'jobseeker' | 'employer'
```

The TypeScript cast provides no runtime validation. The actual value could be any string. The database's `user_role` enum will reject invalid values, but the error path returns a generic "Failed to create user profile" message rather than a clear validation error. If the enum is ever extended (e.g., `'admin'`), any client could register with elevated privileges without an application-level gate.

## Missing role gate on company creation

`createCompany` in `company.ts` authenticates the user and checks whether they already own a company, but never verifies the user's role is `'employer'`. The RLS policy on `companies` for INSERT only enforces `auth.uid() = owner_id`. A job seeker who discovers the server action can create a company record. While this alone doesn't grant employer dashboard access (server actions check company ownership per action), it pollutes data and could create unexpected state if future features rely on company existence as a role indicator.
